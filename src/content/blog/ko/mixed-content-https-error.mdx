SSL 인증서까지 다 발급받고, 도메인도 https로 잘 붙여 놨다. 주소창에 자물쇠도 뜬다. 그런데 막상 페이지에 들어가 보니 스타일이 다 날아갔다. 버튼은 민트초코처럼 깨지고, 이미지는 X 표시만 반짝인다. 개발자 도구를 열어보니 <strong>'Mixed Content'</strong> 경고가 줄줄 뜬다.

사실 https 적용 후 CSS·이미지가 깨지는 문제의 90%는 혼합 콘텐츠(Mixed Content) 때문이다. 페이지는 https로 열리는데, 안에 들어가는 리소스는 여전히 http로 요청하는 상황이다. 여기서 한 번만 정리해 두면, 다음 프로젝트에서는 훨씬 덜 헤맨다.

<br/>

## 1. 혼합 콘텐츠가 뭐길래 CSS·이미지가 깨질까

혼합 콘텐츠는 간단히 말해 <strong>"https 페이지 안에서 http 리소스를 섞어 쓰는"</strong> 상태다. HTML 문서 자체는 암호화된 연결로 받아오는데, 안에 들어가는 이미지·CSS·JS 일부가 암호화되지 않은 http로 로드되는 구조다.

브라우저는 보안 때문에 이걸 점점 더 강하게 막고 있다.

- 크롬, 파이어폭스는 스크립트·CSS처럼 중요 리소스의 http 요청은 <strong>기본적으로 차단</strong>한다.
- 이미지·영상은 예전에는 허용했지만, 최근에는 자동 https 업그레이드 또는 차단이 점점 늘고 있다.

결과적으로

- CSS가 http로 로드되면, 아예 불러오지 못해서 스타일이 모두 풀려 버린다.
- 배경 이미지, `<img>`가 http로 로드되면 깨진 이미지처럼 보이거나, 개발자 도구에서만 에러로 나타난다.

<br/>

## 2. 어디서 http 리소스가 섞였는지 먼저 찾자

해결의 첫 단계는 <strong>"무엇이 http로 로드되고 있는지"</strong> 찾는 것이다.

대표적인 방법은 세 가지다.

1. 브라우저 개발자 도구
   - 크롬: `F12` → Console 탭에서 "Mixed Content" 경고 확인.
   - Network 탭에서 "Scheme" 컬럼을 보면 http/https가 한눈에 보인다.
2. 온라인 검사 도구
   - Why No Padlock 같은 서비스에 URL을 넣어 혼합 콘텐츠를 한 번에 스캔할 수 있다.
3. 정적 코드 검색
   - 프로젝트 전체에서 `http://` 문자열을 검색해 하드코딩된 주소를 찾는다.

워드프레스·CMS 환경에서는, 테마·플러그인·글 내용 안에 예전 `http://도메인` 링크가 잔뜩 남아 있는 경우가 대다수다.

<br/>

## 3. 절대경로 http 링크를 https 또는 프로토콜 상대 경로로 바꾸기

가장 기본적인 해결책이다. HTML, CSS, JS, 템플릿, DB 안에 있는 `http://` 링크를 모두 https 또는 프로토콜 상대 경로로 바꾼다.

예를 들어

- `http://example.com/style.css` → <strong>`https://example.com/style.css`</strong>
- `<img src="http://example.com/img.png">` → <strong>`<img src="https://example.com/img.png">`</strong>
- 또는 `<img src="//example.com/img.png">`처럼 `//`로 시작하는 스킴 상대 URL 사용.

Sucuri와 MDN은 두 가지 패턴을 권장한다.

- 도메인 내 리소스라면 `/img/logo.png` 같은 <strong>루트 기준 상대경로</strong> 사용.
- 외부 리소스라면 `//cdn.example.com/lib.js`처럼 스킴 상대 URL 또는 아예 https로 고정.

국내 블로그 사례에서도 CSS 안의 `background-image: url(http://...)`처럼 하드코딩된 http 경로 때문에 혼합 콘텐츠가 발생하는 경우가 많다고 정리한다.

<br/>

## 4. 상대경로·절대경로 꼬임 점검 (특히 CSS·이미지)

![혼합 컨텐츠 오류로 인해 CSS가 깨져버린 UI 시각화 이미지](/images/broken-css-layout.png)

https로 바꾸고 나니, 이번에는 경로가 꼬여서 또 깨지는 경우도 있다.

자주 나오는 패턴은 이렇다.

- CSS 안에서 `../image/` 같은 상대경로를 썼는데, 실제 배포 경로와 맞지 않는다.
- 템플릿에서 `<img src="/image/test.png">` 대신 <strong>잘못된 경로를 써서 404가 발생</strong>한다.

한 국내 사례에서는, CSS에서 상대경로를 절대경로(`https://도메인/image/...`)로 바꾸고, HTML에서도 동일하게 https가 붙은 절대경로로 정리하면서 문제가 해결되었다고 소개한다. HTTPS 전환 시점에 정적 리소스 구조까지 같이 정리하는 편이 안전하다.

<br/>

## 5. 서버·프록시 레벨에서 http 리소스를 https로 강제 업그레이드

모든 소스를 손으로 고치기 어렵다면, 서버 또는 프록시에서 한 번에 리라이트하는 방법도 있다.

### 5-1. Content-Security-Policy 헤더로 upgrade-insecure-requests

MDN과 구글 web.dev는 `Content-Security-Policy: upgrade-insecure-requests` 헤더를 사용하면 브라우저가 http 리소스를 자동으로 https로 올려준다고 설명한다.

예시 헤더:

```http
Content-Security-Policy: upgrade-insecure-requests
```

이 헤더를 적용하면

- `<img src="http://example.com/img.png">`처럼 남아 있는 http 요청도 브라우저가 <strong>`https://example.com/img.png`로 자동 변환</strong>한다.

단, 실제 서버가 https를 지원해야 한다. 백엔드 서버가 여전히 http만 제공하면, 단순히 업그레이드만으로는 해결되지 않는다.

### 5-2. Nginx/Apache에서 리다이렉트·리라이트 설정

Nginx, Apache 같은 웹 서버나 리버스 프록시에서, 리소스 요청을 https로 리다이렉트하는 규칙을 넣어도 혼합 콘텐츠를 줄일 수 있다.

- Nginx: `return 301 https://$host$request_uri;` 형태의 <strong>HTTP→HTTPS 리다이렉트</strong>.
- Apache: `.htaccess` 또는 vhost 설정에서 `RewriteCond %{HTTPS} off` → `RewriteRule`로 https 강제.

AWS EC2 + Apache + 그누보드 환경에서 SSL 적용 후 CSS가 깨지던 사례도, 리다이렉트 규칙으로 모든 리소스 요청을 https로 바꾸면서 해결되었다고 기록한다.

<br/>

## 6. Cloudflare 등 CDN에서 자동 HTTPS Rewrites 활용

Cloudflare 같은 CDN을 쓰고 있다면, 대시보드에서 몇 가지 옵션만 켜도 혼합 콘텐츠가 상당 부분 정리된다.

Cloudflare 공식 문서와 2025년 가이드 영상에서는 다음을 권장한다.

- SSL/TLS → Edge Certificates에서 <strong>Automatic HTTPS Rewrites 무조건 활성화</strong>.
- 필요하다면 Always Use HTTPS(모든 요청을 https로 리다이렉트)도 함께 설정.

이 기능은 HTML 안에 있는 `http://` 리소스 URL을 자동으로 https 버전으로 바꾸려고 시도한다. 워드프레스라면 Cloudflare 플러그인에서 같은 옵션을 켤 수 있다.

단, 원본 서버(origin)가 https를 제대로 지원해야 한다. Cloudflare 문서에서도 "모든 리소스를 https로 로드할 수 있다는 전제"가 필요하다고 강조한다.

<br/>

## 7. 워드프레스·CMS에서 DB 전체 HTTP → HTTPS 일괄 변환

워드프레스, 우커머스 같은 CMS에서는 이미지, 첨부파일, 내부 링크가 데이터베이스 안에 http로 저장되어 있는 경우가 많다.

이때는

- WP-CLI 또는 Better Search Replace 같은 플러그인으로 <strong>`http://도메인` → `https://도메인` 일괄 치환</strong>.
- 필요하다면 직후에 썸네일 재생성, 캐시 플러그인 캐시 삭제.

워드프레스 커뮤니티에서는 Better Search Replace로 DB 전체를 치환하고, "SSL Insecure Content Fixer" 같은 플러그인으로 남은 혼합 콘텐츠를 마무리 정리하라고 조언한다. 이때도 캐시를 꼭 비워야 변경 사항이 바로 반영된다.

<br/>

## 8. 외부 리소스가 https를 지원하지 않는 경우

가끔은 근본적으로 해결이 안 되는 경우도 있다. 외부에서 끌어오는 이미지, 스크립트, 폰트 서버가 https를 지원하지 않는 경우다.

이때 선택지는 세 가지 정도다.

- 해당 리소스를 <strong>직접 서버에 내려받아</strong>, 내 도메인(https)에서 서빙한다.
- https를 지원하는 <strong>다른 CDN/대체 라이브러리로 교체</strong>한다.
- 정말 대안이 없다면, 해당 리소스를 <strong>사용하지 않는다</strong>.

Sucuri와 여러 보안 문서는, 혼합 콘텐츠를 남겨두는 것 자체가 공격 표면을 넓힌다고 경고한다. 눈에 잘 안 보이는 이미지 한 장이라도, http로 남겨두는 건 장기적으로 좋지 않다.

<br/>

## 9. 캐시·브라우저·CDN까지 싹 비우고 다시 확인하기

모든 수정을 적용한 뒤에도 CSS·이미지가 여전히 깨져 보인다면, 그건 <strong>캐시 때문</strong>이다.

- Cloudflare/CDN 캐시에서 이전 http 리소스 응답을 계속 내보내고 있을 수 있다.
- 브라우저 캐시, 서비스 워커가 오래된 정적 파일을 들고 있을 수 있다.

그래서 마지막에는

- CDN/리버스 프록시 <strong>캐시 전체 Purge</strong>
- 워드프레스·CMS <strong>캐시 플러그인 비우기</strong>
- 브라우저 시크릿 모드나 강력 새로고침(Ctrl+F5)으로 재확인

이 세 단계를 거친 뒤에야, 진짜로 문제가 해결되었는지 판단하는 게 좋다.

<br/>

## 10. 실전 체크리스트

정리하면, https 전환 후 CSS·이미지가 깨질 때는 아래 순서로 보면 대부분 해결된다.

1. <strong>콘솔에서 Mixed Content 경고 확인</strong>
   - 어떤 리소스가 http로 요청되고 있는지 목록을 뽑는다.
2. <strong>코드·DB에서 `http://도메인` 검색</strong>
   - HTML, CSS, JS, 템플릿, DB 내용에서 http 링크를 https 또는 상대경로로 수정한다.
3. <strong>서버/헤더 설정 보완</strong>
   - `Content-Security-Policy: upgrade-insecure-requests` 헤더 검토.
   - Nginx/Apache의 HTTP→HTTPS 리다이렉트, 정적 리소스 경로 점검.
4. <strong>CDN·플러그인 활용</strong>
   - Cloudflare Automatic HTTPS Rewrites, 워드프레스용 혼합 콘텐츠 플러그인 등 활용.
5. <strong>외부 리소스 교체</strong>
   - https 미지원 외부 리소스는 직접 호스팅 또는 대체 리소스로 교체.
6. <strong>모든 캐시 삭제 후 재테스트</strong>
   - CDN, 애플리케이션, 브라우저 캐시를 모두 비우고 다시 확인.

한 번 제대로 정리해 두면, 이후 프로젝트에서 SSL 적용은 "자물쇠 켜고 끝"에 가까워진다.
