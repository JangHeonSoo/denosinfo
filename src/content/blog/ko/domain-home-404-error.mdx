도메인을 겨우 연결했다. DNS도 맞게 설정했고, 서버도 잘 뜨는 것 같다. 기대하면서 `https://내도메인.com`에 접속해 본다. 그런데 첫 화면이 404다. <strong>"Page Not Found"</strong>.

사실 이 케이스는 생각보다 단순한 문제에서 많이 나온다. 도메인과 서버가 각각 제 역할을 하는데, 둘 사이 연결 고리가 어긋난 상태다. 그래서 여기서는 <strong>"도메인은 연결됐는데 첫 화면이 404가 뜨는"</strong> 상황에서 점검해야 할 대표 원인과 해결 방법을 정리한다.

<br/>

## 1. 도메인이 아예 다른 곳을 가리키는 경우 (DNS·네임서버 문제)

가장 먼저 확인해야 할 건, 도메인이 진짜 내 서버를 가리키는지다. 도메인이 잘 연결된 줄 알았는데, 알고 보니 예전 호스팅 업체나 엉뚱한 IP를 보고 있는 경우가 많다.

대표적인 상황은 이렇다.

- 도메인을 새 호스팅으로 옮겼는데, 네임서버는 아직 옛 호스트를 가리키고 있다.
- A 레코드가 예전 서버 IP를 계속 바라보고 있다.
- 서브도메인(www, api 등) 레코드는 바꿨지만 루트 도메인(@) 레코드는 그대로인 경우.

호스팅 튜토리얼에서는, 마이그레이션 후 `index.php`, `.htaccess` 등 필수 파일이 있는 새 서버를 향하도록 <strong>네임서버와 DNS를 반드시 업데이트</strong>해야 한다고 강조한다. 도메인이 잘못된 서버를 가리키면, 그 서버 입장에서는 해당 경로에 페이지가 없으므로 404를 돌려보내는 게 정상이다.

체크 포인트는 간단하다.

- `nslookup 내도메인.com`으로 실제로 어떤 IP를 가리키는지 확인한다.
- 그 IP가 내가 기대한 서버의 공인 IP와 같은지 비교한다.

<br/>

## 2. 홈 페이지가 없거나, 다른 경로에 있는 경우

도메인은 제대로 연결되어 있는데, 웹 서버가 보여줄 "첫 페이지"를 찾지 못해서 404를 내보내는 경우도 많다.

예를 들어 이런 경우다.

- 웹 루트 디렉터리에 `index.html`이나 `index.php` 같은 기본 문서가 없다.
- 실제 홈 페이지는 `/home`, `/main` 같은 경로에 있는데, 루트(`/`)에 별도 라우트나 리다이렉트 설정이 없다.
- CMS(HubSpot, Squarespace 등)에서 홈 페이지가 지정되지 않았거나 비활성화되어 있다.

HubSpot 커뮤니티에서는 도메인 접속 시 404가 뜨는 문제에 대해, 홈 페이지의 슬러그를 비워서 루트(`/`)에 매핑한 뒤 다시 발행하라고 안내한다. Squarespace에서도 도메인이 연결되었는데 404가 나오면, 사이트 내에서 실제로 <strong>"Set as homepage"가 켜진 페이지가 있는지</strong> 확인하라는 답변이 반복된다.

직접 서버를 운영한다면, 다음을 확인해 봐야 한다.

- 웹 루트 디렉터리(예: `/var/www/html`, `/home/계정/public_html`)에 기본 문서가 있는지.
- Nginx/Apache 설정에서 `index index.html index.php;` 같은 <strong>기본 인덱스 설정</strong>이 되어 있는지.

<br/>

## 3. 웹 서버의 DocumentRoot가 잘못 지정된 경우

도메인도 맞고 서버도 켜져 있는데, 웹 서버 설정이 엉뚱한 폴더를 가리키면 그 폴더 기준으로 404가 발생한다.

예를 들어, 실제 파일은 `/home/user/project/build`에 있는데, Apache의 `DocumentRoot`는 `/var/www/html`로 되어 있는 식이다. 이 디렉터리에 아무 파일도 없으면, 루트(`/`)로 접근했을 때 404가 뜬다.

Nginx 튜토리얼에서는 `root`와 `alias` 차이 때문에 404가 나는 사례도 소개한다.

- `root`는 `location`의 경로를 통째로 붙여서 파일을 찾는다.
- `alias`는 `location` 이후의 경로만 붙여서 찾는다.

설정이 꼬이면 Nginx가 `/usr/share/nginx/html/index.html` 대신 `/usr/share/nginx/html/context-path/index.html` 같은 엉뚱한 곳을 뒤지게 되고, 결국 404를 반환한다.

따라서 다음을 확인해야 한다.

- Apache: `<VirtualHost>` 블록의 `DocumentRoot`가 <strong>실제 프로젝트 경로와 일치하는지</strong>.
- Nginx: `server { root ... }`와 `location` 블록, `alias` 사용 여부가 올바른지.

<br/>

## 4. 가상호스트/서버블록(vhost) 설정이 도메인과 안 맞는 경우

서버 한 대에 여러 도메인을 올려 쓰는 경우, 가상호스트 설정이 꼬이면 엉뚱한 사이트 설정이 적용되면서 404가 뜰 수 있다.

예를 들면 이런 상황이다.

- `ServerName`, `server_name`이 실제 도메인과 다르게 설정되어 있다.
- 공용 `default` 서버블록이 모든 요청을 받아 404를 내보내고 있다.
- 패널(DirectAdmin 등)에서 서버 호스트네임과 실제 도메인 설정이 충돌한다.

실제 사례에서, nginx+apache 조합 환경에서 딱 한 도메인만 404가 뜨던 문제는 DirectAdmin 설정에서 서버 호스트네임과 도메인 이름이 충돌한 것이 원인이었다. 같은 서버에 여러 도메인을 연결했다면, 반드시 다음을 확인해야 한다.

- Apache: 각 `<VirtualHost>`가 올바른 `ServerName`, `ServerAlias`, `DocumentRoot`를 갖고 있는지.
- Nginx: `server_name 내도메인.com www.내도메인.com;`처럼 정확히 명시되어 있는지.

<br/>

## 5. 애플리케이션 라우팅·리라이트(rewrite) 설정 문제

![네트워크 라우팅 오류 추적 시각화 이미지](/images/network-routing-issue.png)

정적 HTML 한두 개만 있는 사이트라면 상관없지만, Laravel, Spring, React SPA 등은 라우팅과 리라이트 설정이 틀어지면 루트에서 404가 뜬다.

예를 들어 Laravel+Nginx 환경에서, `/` 요청이 `index.php`로 전달되지 않으면 모든 라우트가 404로 처리될 수 있다. 실제로는 `https://domain.com/index.php/some-route`로 요청하면 잘 되는데, `https://domain.com/some-route`로는 404가 나는 패턴이다.

이럴 때는 다음을 확인한다.

- `.htaccess` 또는 Nginx 설정에 <strong>프레임워크용 rewrite 규칙</strong>이 제대로 들어가 있는지.
- 퍼머링크 구조를 변경했다면, 워드프레스 기준으로 설정 → 고유주소에서 한 번 저장해 `.htaccess`를 재생성했는지.

<br/>

## 6. 캐시·CDN이 오래된 404를 계속 내보내는 경우

도메인과 서버 설정을 고쳤는데도 첫 화면이 계속 404라면, 중간에 끼어 있는 캐시 계층을 의심해야 한다.

- Cloudflare, Fastly 같은 CDN에서 예전 404 응답을 캐싱하고 있는 경우.
- 브라우저 캐시, 서비스 워커가 오래된 리소스를 계속 서빙하는 경우.

호스팅 가이드에서는 404 관련 설정을 바꾼 뒤, <strong>CDN 캐시를 강제로 퍼지(purge)</strong>하고 브라우저 캐시·쿠키를 지운 상태에서 다시 테스트하라고 조언한다.

<br/>

## 7. 플랫폼 특유의 "홈페이지 지정" 이슈 (노코드/호스팅 빌더)

Wix, Squarespace, Google Sites, HubSpot 같은 빌더 기반 서비스에서는, 도메인 연결과 "홈 페이지 지정"이 분리되어 있는 경우가 많다.

실제 사례를 보면:

- 도메인 상태는 "Connected" 또는 "Success"인데, 홈 페이지로 지정된 페이지가 없다.
- 페이지는 만들어져 있지만 "비공개" 또는 "비활성" 상태라 루트에서 404가 뜬다.
- `www.domain.com`은 잘 열리는데, `domain.com`(naked domain)은 404가 뜨는 구조다.

이런 플랫폼을 쓴다면, 보통 다음 메뉴를 한 번씩 훑어보면 해결된다.

- 사이트 설정 → 도메인 → 기본 도메인/루트 도메인 연결 상태
- 페이지 설정 → <strong>특정 페이지를 "홈 페이지"로 지정</strong>하는 옵션
- 도메인 공급업체에서 `domain.com` → `www.domain.com` 리다이렉트 설정 여부

<br/>

## 8. 실제로는 404가 아니라, 다른 에러를 404로 감싼 경우

가끔은 진짜 원인이 500, 502, 권한 오류인데, 애플리케이션이 <strong>커스텀 404 페이지로 감싸서</strong> 보여주는 경우도 있다. 겉으로만 404로 보일 뿐이다.

이 경우에는 브라우저 개발자 도구의 네트워크 탭이나 서버 로그를 확인해야 한다.

- Nginx/Apache 에러 로그에 403, 500, 502 등의 에러가 찍히는지.
- 애플리케이션 로그에 예외가 발생했는지.

진짜 HTTP 상태 코드가 무엇인지 확인하면, 문제 방향이 완전히 달라지기도 한다.

<br/>

## 9. 빠르게 점검해 보는 순서

원인이 많아 보여도, 순서대로만 보면 생각보다 빨리 잡힌다.

1. <strong>DNS 확인</strong>
   - `nslookup 내도메인.com`으로 현재 A 레코드 IP 확인.
   - 기대한 서버 IP와 다르면, DNS·네임서버부터 수정.
2. <strong>서버 루트·홈페이지 확인</strong>
   - 웹 루트 디렉터리에 `index.html`/`index.php`가 있는지 확인.
   - Nginx/Apache의 `root`/`DocumentRoot`, `index` 설정 점검.
3. <strong>라우팅/리라이트 확인</strong>
   - 프레임워크 사용 시 `.htaccess` 또는 Nginx rewrite 규칙 점검.
   - 워드프레스라면 고유주소 설정 저장으로 .htaccess 재생성.
4. <strong>빌더/플랫폼 설정 확인</strong>
   - 사용 중인 플랫폼에서 홈 페이지가 지정되어 있는지.
   - `www`와 루트 도메인이 모두 연결/리다이렉트 되는지.
5. <strong>캐시 삭제</strong>
   - CDN 캐시 퍼지, 브라우저 캐시 삭제 후 재접속.
