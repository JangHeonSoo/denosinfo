로컬에서는 서버가 잘 열린다. `localhost:8080`으로 들어가면 화면이 반듯하게 뜬다. 그런데 집 밖, 회사, 휴대폰 LTE에서 접속하면 타임아웃이나 연결 실패만 돌아온다. 서버는 살아 있는데, 세상과 단절된 섬이 된 느낌이다.

사실 이런 상황은 생각보다 자주 생긴다. 웹 서버, 게임 서버, 개발 중인 API, 뭐든 간에. 그래서 여기서는 <strong>"서버 프로세스는 떠 있는데 외부에서 접속이 안 되는"</strong> 상황에서 많이 나오는 원인을 한 번에 정리한다.

## 1. 서버가 127.0.0.1(로컬호스트)만 리슨 중인 경우

가장 흔한, 그리고 가장 허무한 원인이다. 서버가 내부 루프백 주소인 <strong>127.0.0.1</strong>에만 바인딩되어 있어서, 같은 머신에서는 접속이 되지만 외부 IP로는 절대 접속이 안 되는 경우다.

Stack Overflow에서도 "로컬에서는 잘 되는데, 다른 컴퓨터에서는 접속이 안 된다"는 질문의 상당수가 이 문제로 끝난다. 서버가 리슨할 수 있는 IP는 보통 세 가지 패턴이다.

- <strong>127.0.0.1</strong>: 오직 로컬 컴퓨터에서만 접속 가능하다.
- <strong>192.168.x.x, 10.x.x.x</strong> 같은 사설 IP: 같은 내부망(LAN)에서만 접속 가능하다.
- <strong>0.0.0.0</strong>: 모든 인터페이스에서 접속 가능하다.

서버 설정에서 바인딩 주소가 <strong>127.0.0.1</strong>로 되어 있으면, 외부 접속은 원천 봉쇄다. 스프링 부트, Node.js, Python 서버 등 대부분의 프레임워크에서 <strong>0.0.0.0</strong>으로 바꿔주면 외부에서도 접속할 수 있다.

<br/>

## 2. 서버 내부 방화벽에서 포트를 막고 있는 경우

애플리케이션은 잘 떠 있지만, OS 차원의 방화벽이 외부에서 오는 패킷을 차단하는 케이스다. 리눅스라면 <strong>iptables</strong>나 <strong>ufw</strong>, 윈도우라면 Windows Defender 방화벽이 대표적이다.

국내 트러블슈팅 글에서도 "inbound 정책은 기본 차단, 필요한 포트만 화이트리스트로 허용"하는 구조를 반복해서 강조한다. 서버에서 포트가 떠 있는지 확인하려면 다음 명령으로 체크할 수 있다.

- 리눅스: `sudo netstat -tulpn | grep 포트` 또는 `ss -tulpn | grep 포트`
- 윈도우: `netstat -ano | findstr 포트`

포트가 LISTEN 상태인데 외부 접속이 안 된다면, 다음을 의심해야 한다.

- 서버 방화벽에서 해당 포트 inbound 허용 규칙이 없는지.
- 보안 그룹(클라우드의 경우)에서 0.0.0.0/0 또는 필요한 IP 대역에 대해 허용이 안 되어 있는지.

실무 가이드에서는 서버 방화벽에 규칙을 추가한 뒤, 외부에서 `telnet 서버IP 포트`로 접속 테스트를 하라고 안내한다.

<br/>

## 3. 네트워크 방화벽·라우터에서 막힌 경우 (포트 포워딩, 인바운드 정책)

![미래 지향적인 로봇 팔이 방화벽의 인바운드/아웃바운드 정책을 세밀하게 조정하는 시각화](/images/server_access_firewall_robotic_hand_1771459334384.png)

서버 앞단에 공유기나 네트워크 방화벽이 있으면, 거기서 한 번 더 막힐 수 있다. 집에서 쓰는 NAS, 홈 서버, 사무실 내부 서버에서 특히 자주 터지는 문제다.

대표적인 케이스는 이렇다.

- 공유기에서 포트 포워딩을 안 했거나 잘못 설정했다.
- 방화벽 장비에서 해당 포트의 inbound 정책이 막혀 있다.
- WAN IP는 맞는데, 그 IP에서 서버로 가는 라우팅이나 NAT 설정이 잘못되었다.

국내 네트워크 트러블슈팅 글에서는 <strong>"서버와 외부 인터넷 사이에 방화벽이 존재하면, 방화벽 inbound 정책에서 서버 IP와 서비스 포트를 명시적으로 허용해야 한다"</strong>고 정리한다. Let’s Encrypt/ACME 인증 과정에서 자주 보이는 `Timeout during connect (likely firewall problem)` 에러 역시, 포트 80/443에 대한 방화벽 또는 포트 포워딩 문제로 발생한다고 문서에 적혀 있다.

외부에서 접속이 안 되면, 우선

- 공유기 포트 포워딩 테이블 확인
- 방화벽 장비 inbound 룰에서 서버 IP+포트 허용 여부 확인

이 두 가지를 먼저 봐야 한다.

<br/>

## 4. 공인 IP, 사설 IP, 도메인·DNS 설정 혼선

서버는 잘 뜨고, 내부망에서도 잘 접속된다. 그런데 도메인으로 접속하면 안 들어가거나, 집 밖에서는 안 된다. 이런 경우 IP와 DNS가 제각각인 상황이 많다.

자주 나오는 혼선은 다음과 같다.

- 내부에서는 <strong>192.168.x.x:8080</strong>으로 잘 들어가는데, 외부에서도 그 주소로 접속하려고 한다.
- 도메인이 예전 IP를 가리키고 있어서, 엉뚱한 서버로 가거나 타임아웃이 난다.
- DDNS를 쓰는데, 공유기의 공인 IP가 바뀐 뒤로 업데이트가 안 됐다.

Stack Overflow 답변에서는 <strong>"로컬 주소(192.168.x.x)는 전 세계에 수천 대가 같은 주소를 쓰므로, 외부에서 접속할 수 없다"</strong>는 점을 반복해서 설명한다. 외부에서 접속하려면

- 도메인 → 공인 IP → 라우터 → 서버 사설 IP → 서버 포트

이 흐름이 정확히 맞아떨어져야 한다.

<br/>

## 5. 클라우드 보안 그룹(SG) 설정 누락

AWS, GCP, Azure 같은 클라우드 환경에서는 OS 방화벽 말고도 한 단계 더 방화벽이 있다. 바로 <strong>보안 그룹(Security Group)</strong> 또는 네트워크 보안 규칙이다.

여기에서 인바운드 규칙을 열어주지 않으면, 서버 프로세스가 떠 있어도 외부에서는 포트가 완전히 닫힌 것처럼 보인다. 예를 들어, AWS EC2에서는 다음과 같은 설정이 필요하다.

- 인바운드 규칙에 <strong>TCP 80</strong> 또는 <strong>TCP 443</strong>을 0.0.0.0/0에 대해 허용
- 특정 관리자용 포트는 고정 IP에 대해서만 허용

실제 Q&A와 가이드에서도 "서버가 잘 뜨는데 외부 접속이 안 된다"는 문의의 상당수가 보안 그룹 인바운드 규칙 미설정으로 결론난다.

<br/>

## 6. 애플리케이션이 다른 포트·다른 인터페이스에서 떠 있는 경우

개발 환경에서 자주 나오는 실수다. 문서에는 8080 포트라고 적혀 있는데, 실제로는 3000 포트에서 떠 있다든지, 혹은 IPv6 주소에서만 리슨한다든지 하는 경우다.

이럴 때는 다음을 확인하면 된다.

- 서버 내부에서 `curl http://localhost:포트`로 실제 리슨 중인 포트를 확인
- <strong>netstat</strong> 또는 <strong>ss</strong>로 어떤 IP/포트 조합에 바인딩되어 있는지 확인

리눅스 트러블슈팅 글에서는, 애플리케이션 설정이 꼬였는지 확인하기 위해 Python의 `python3 -m http.server 포트` 같은 간단한 웹 서버를 띄워 외부 접속 여부를 테스트해 보라고 권장한다. 이 테스트 서버는 잘 접속되는데, 원래 애플리케이션만 안 된다면, 문제는 네트워크가 아니라 애플리케이션 설정이다.

<br/>

## 7. TCP 포트는 열려 있는데, DNS가 엉켜 있는 경우

포트도 열려 있고, 보안 그룹도 다 열어놨는데 도메인으로는 접속이 안 되는 경우가 있다. 이때는 DNS가 가장 의심스럽다.

대표 징후는 다음과 같다.

- IP로 접속하면 잘 되는데, 도메인으로 하면 타임아웃 또는 다른 사이트가 열린다.
- <strong>ping 도메인</strong> 결과가 기대한 IP와 다르게 나온다.
- <strong>nslookup</strong>이 timeout을 내거나 엉뚱한 레코드를 보여 준다.

네트워크 포럼에서는 이런 상황에서 "DNS 포트(53)가 방화벽에 막혀 있거나, 요청은 나가는데 응답이 돌아오는 경로가 다르게 설정되어 중간에 유실되는 경우"도 있다고 설명한다. 도메인 접속 문제가 의심되면

- A 레코드가 최신 공인 IP를 가리키는지
- TTL이 너무 길게 잡혀 있지는 않은지
- 내부 DNS와 외부 DNS가 서로 다른 값을 주지 않는지

를 함께 확인해야 한다.

<br/>

## 8. NAT, 이중 NAT, 헤어핀 NAT 문제

집이나 사무실에서 흔히 만나는 복잡한 케이스다. 공유기가 두 대 이상이며, 사설 IP 대역이 여러 개 섞여 있는 구조에서 외부 접속이 꼬인다.

- <strong>이중 NAT(Double NAT)</strong> 환경이라 공인 IP → 공유기1 → 공유기2 → 서버로 가는 경로 어딘가에서 포트 포워딩이 끊긴다.
- 같은 내부망에서 도메인으로 접속하면, 공유기가 자기 자신을 어떻게 돌려보낼지 몰라서 접속이 안 되는 경우도 있다.

일부 라우터는 <strong>헤어핀 NAT(또는 NAT loopback)</strong>를 지원해야 내부에서 외부 도메인으로 자기 서버에 접속할 수 있다고 설명한다. 지원하지 않는 라우터에서는, 내부에서는 hosts 파일이나 내부 DNS 서버를 따로 두라고 네트워크 커뮤니티에서 조언한다.

<br/>

## 9. 클라이언트·ISP 쪽에서 막는 경우

서버만 계속 의심하다가, 막상 원인이 클라이언트나 ISP 쪽 필터링인 경우도 있다.

예를 들어

- 회사 네트워크에서 22번 포트(SSH), 3389(원격 데스크톱) 등 특정 포트를 inbound 차단해 둔 경우
- 일부 통신사/국가에서 특정 포트를 기본 차단하거나, HTTPS가 아닌 트래픽을 프록시로 통제하는 경우

이럴 때는 다른 네트워크(휴대폰 테더링, 다른 ISP, VPN 등)에서 접속해 보거나, 클라이언트 쪽 방화벽·보안 솔루션을 잠시 끄고 테스트해 보는 것이 도움이 된다.

<br/>

## 10. 진단 순서 예시

![터미널과 네트워크 맵을 대조하며 차례대로 원인을 분석하는 고해상도 워크스페이스 사진](/images/server_access_diagnosis_engineer_desk_1771459353541.png)

원인 후보가 많아서 헷갈리기 쉽다. 그래서 보통은 아래 순서로 하나씩 잘라 나가는 게 빠르다.

1. <strong>서버 내부에서 확인</strong>
   - `curl http://localhost:포트`로 애플리케이션이 정상 응답하는지 확인한다.
   - <strong>netstat/ss</strong>로 어떤 IP에 바인딩되어 있는지 확인한다(127.0.0.1인지, 0.0.0.0인지).
2. <strong>같은 내부망(LAN)에서 확인</strong>
   - 다른 PC나 휴대폰 와이파이로 `http://서버내부IP:포트`에 접속해 본다.
   - 이 단계에서 안 되면, 서버 방화벽·내부 라우팅부터 의심한다.
3. <strong>외부망에서 확인</strong>
   - 휴대폰 LTE, 다른 네트워크에서 공인 IP 또는 도메인으로 접속해 본다.
   - 안 되면 공유기 포트 포워딩, 외부 방화벽, 보안 그룹을 확인한다.
4. <strong>DNS 확인</strong>
   - <strong>ping 도메인, nslookup 도메인</strong>으로 실제 어떤 IP를 가리키는지 확인한다.

이 순서대로만 체크해도, 대부분의 "서버는 떠 있는데 외부에서 접속이 안 되는" 문제는 원인을 좁힐 수 있다.
